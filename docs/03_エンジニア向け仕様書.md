# 定跡ツリー — 仕様書（エンジニア向け）

> アプリの機能・データモデル・技術構成とその選定理由をまとめた、開発者向けのリファレンスです。
> 「作って渡す」ための指示書は別途 `01_claude_code_実装指示書.md` を参照。
> アプリ名「定跡ツリー」は仮称です。

---

## 1. 概要

将棋の定跡研究と反復学習のための個人用 Web アプリ。定跡の手順を作り、仕掛けが成立する局面を記録し、それを覚えるための問題集を作る——という一連の流れを、**1つの木構造**に統合して扱う。スマホ・PC の両方から使え、クラウド同期により端末をまたいで同じデータを扱える。

---

## 2. 目的と設計思想

### 2.1 3つの目的
1. オリジナルの定跡手順を作成できる
2. 仕掛けが成立する局面を記録できる
3. 上記を記憶するための問題集を作成できる

### 2.2 「すべてはひとつの木」という中核
一見別々のこれらの機能は、データとして見るとすべて同じ形に還元できる。

- **局面ノード**: ある盤面の状態（SFEN で一意に表せる）
- **手（枝）**: 局面 A から局面 B へつなぐ枝
- **定跡ツリー**: 初期局面を根として手が枝分かれしていく木。枝に戦法名・分岐名（急戦／持久戦、45歩早仕掛け…）のラベルが付く
- **棋譜（分岐つき）**: 木の中の一本の経路（＋変化）。構造は木と同じ
- **問題**: 木のあるノードを指して「この局面で正解の手は？」と問う、木の上に貼るフラグ

つまりアプリの核は「局面ノードを手でつないだ木＋注釈」ひとつであり、3つの機能はその木に対する **編集 / 可視化（樹形図）/ 出題** という異なるビューにすぎない。この一貫した基盤により、樹形図・棋譜・問題集がバラバラに実装されて食い違う事故を防ぐ。

なお、元要件にあった「スナップショット」機能は、この構造では**木のノードそのものが実質スナップショット**になる（任意ノードで分岐を足せる＝そこから編集できる）ため、独立機能としては設けない。

## 3. 機能仕様

### 3.1 盤面と駒操作
- 9×9 盤と、先手・後手の持ち駒台を描画する。
- 駒をタップで選択し、移動先マスをタップして着手する。
- 成れる手では成／不成を選択する。行き所のない駒になる手は自動的に成りへ確定する（強制成り）。
- 持ち駒を打てる。

### 3.2 合法手ハイライト・反則判定
- 選択中の駒（または打とうとしている駒種）について、移動可能マス・打てるマスをハイライトする。
- 合法手はルールエンジン（`shogiops`）が生成する。以下の反則は合法手生成の時点で除外される：
  - 二歩
  - 行き所のない駒（歩・香の最上段、桂の上2段）
  - 打ち歩詰め
  - 王手放置
- ルール判定を自前実装しないことで、実装の沼を回避する。

### 3.3 木の編集（定跡ツリー）
- 現在ノードで手を指すと、既存の子ノードに一致すればそのノードへ移動し、一致しなければ新規子ノードを作成する。この「新規子ノード作成」が**分岐**にあたる。
- 1つの局面から複数の手が伸びれば、それが戦法・変化の分岐になる。
- 任意のノードに、戦法名・分岐名（枝のラベル）とコメントを付けられる。

### 3.4 樹形図
- 木全体を俯瞰する樹形図を描画する。ノードをタップすると、盤面をその局面へ移動する。
- 盤面ビューと樹形図ビューを切り替えられる。
- 分岐点ごとにワンクリックでアクセスできる、という元要件はこの樹形図で満たす。

### 3.5 問題集（ノートブック・出題・解説・再挑戦・成績）
- **ノートブック**: 問題をまとめる器。作成・一覧・編集・削除（CRUD）ができる。種別ごとに分けて管理する用途。
- **出題形式**: 「1手を当てる」。ある局面を提示し、正しい次の一手を回答させる。
- **正解の定義**: 問題ごとに選べる。
  - 単一手を正解とする（例：この局面では45歩の一手だけ正解）
  - 木にある子の手ならどれでも正解とする（分岐が存在すること自体を覚える問題）
- **解説**: 回答後に解説を表示する。解説はフリーテキスト（符号を含められる）に加えて、**局面再生の起点ノード**を持ち、そこから木の経路をたどってミニ盤でステップ再生できる。符号だけでも成立するが、局面再生まで行えるのが望ましい形。
- **再挑戦**: 演習セッション内で、間違えた問題をプールに戻して再挑戦できる。
- **成績履歴**: 挑戦回数・正誤を記録する。この履歴を持たせておくことで、後から SRS（間隔反復）を追加できる。

## 4. データモデル

「1手＝1ノード」の木。各ノードは、自分に至る手・その局面（SFEN）・親への参照を持ち、分岐は「同じ親を持つ複数ノード」で表す。棋譜・問題・ノートブックはこの木を参照する形で乗る。

### 4.1 エンティティ
| エンティティ | 役割 | 主な項目 |
|---|---|---|
| trees | 定跡ツリー | name, description |
| nodes | 局面ノード（1手=1ノード） | parent_id, move_usi, sfen, branch_label, comment |
| notebooks | ノートブック | name, description |
| problems | 問題（1手当て） | node_id, answer_move_usi, accept_any_child, explanation_text, explanation_from_node_id |
| problem_reviews | 成績履歴／復習状態 | attempts, correct_count, wrong_count, last_reviewed_at, （SRS用: srs_ease, srs_interval_days, srs_due_at） |

### 4.2 設計上のポイント
- **SFEN を局面の一意キー**として使い、ノードの同一性・重複検出に利用する。
- ノードが「自分に至る手」を保持することで、木＝棋譜の分岐構造をそのまま表現できる。これはルールライブラリ `shogiops` の姉妹実装 `tsshogi` が採る「1手＝1ノード（next/prev/branch）」の考え方とも整合する。
- 問題は木のノードを指すだけの薄い存在にし、盤面や手の情報は木に一元化する（重複・不整合を避ける）。
- 成績・SRS 用の列は最初から用意しておき、MVP ではプール方式のみ実装、後から SRS ロジックを足せるようにする（作り直し不要）。
- 保存先はクラウド（Postgres）だが、**データモデル自体は保存先に依存しない**。ローカル保存に切り替えても同じ構造で成立する。

DDL の詳細は `01_claude_code_実装指示書.md` の「3. データモデル」を参照。

## 5. 技術スタックと選定理由

| 領域 | 採用 | 選定理由 |
|---|---|---|
| フロント | React + TypeScript + Vite | 既存スキル（React/TS）をそのまま活かせる。Vite は軽量で PWA 化しやすい |
| プラットフォーム | PWA（Web アプリ） | 1コードでスマホ・PC 両対応。ブラウザで動き、インストールもできる。ネイティブより両対応の手間が少ない |
| 将棋ルール | `shogiops`（GPL-3.0） | 合法手・打ち駒の生成、妥当性チェック、SFEN/USI 入出力を明示的に備える。lishogi を支える実績。ルール判定の自前実装を避けられる |
| 盤 UI | 自作 React コンポーネント | 9×9 グリッド＋駒でハイライト・タップ操作まで思い通りに作れる。得意分野。省力化したい場合は `shogiground` も選択肢 |
| バックエンド | Supabase（Postgres + Auth + RLS） | リレーショナルなデータモデルに素直に合う。認証と「自分のデータだけ見える」制御（RLS）が標準で揃う。サーバー運用不要、個人利用は無料枠で十分 |
| 同期・保存 | Supabase（クラウドを正とする） | 単一ユーザーが複数端末で使う前提。last-write-wins で十分 |
| ホスティング | Vercel / Netlify / Cloudflare Pages | 静的 PWA ＋ BaaS のため無料枠で完結 |

**なぜクラウド＋ログインなのか**: スマホと PC の両方から同じデータを使う（データ共有）ためには、端末内保存では不十分で、両端末から見えるクラウドに置く必要がある。そのために簡単な認証（ログイン）を入れ、「あなたのデータ」を端末をまたいで追従させる。

**将来 KIF 入出力が必要になったら**: `tsshogi`（MIT）を足す。KIF/KI2/CSA/JKF/SFEN/USI の入出力に対応し、棋譜を木構造で扱えるため、本アプリのモデルとも親和性が高い。

## 6. 非機能・制約

- **オフライン**: service worker によるアプリシェルのキャッシュまで。データ操作はオンライン前提。本格的なオフライン同期・競合解決は対象外。
- **同期の競合**: 単一ユーザー・複数端末を想定し last-write-wins。複数人での同時編集は想定しない。
- **セキュリティ**: RLS により、ユーザーは自分の行にのみアクセスできる。
- **コスト**: 個人利用の範囲では Supabase・ホスティングともに無料枠で運用可能。
- **ライセンス**: `shogiops` は GPL-3.0-or-later。個人アプリでの利用は問題ないが、配布形態を変える場合は留意する。

## 7. 将来拡張（ロードマップ）

- **SRS（間隔反復）**: `problem_reviews` の SRS 用列を使い、忘却曲線に沿った復習日スケジューリングと出題を追加。定着度が大きく上がる。
- **局面編集（空盤配置）**: 任意局面を初手からの手順を経ずに直接組めるようにする。合法性チェック・持ち駒設定の実装が必要。
- **KIF/CSA 入出力**: `tsshogi` を導入し、既存棋譜の取り込み・書き出しに対応。
- **手順を当てる出題**: 1手当てに加え、寄せや手筋を最後まで指し継ぐ形式。
- **検討・エンジン連携**: USI エンジンとの接続（大きな拡張）。

## 8. 用語

- **SFEN**: 局面を表す文字列表現（USI プロトコルで用いられる、チェスの FEN の将棋版）。持ち駒の情報を含む。
- **USI**: 将棋の思考エンジン用プロトコル。手の座標表記（例 `7g7f`）や SFEN を用いる。
- **定跡**: 序盤で研究・確立された指し手の手順。本アプリでは木として蓄積する。
- **SRS**: Spaced Repetition System（間隔反復学習）。忘れた頃に復習を出す仕組み。
